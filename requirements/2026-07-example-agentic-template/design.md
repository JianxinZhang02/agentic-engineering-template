# Design: Agentic Engineering 模板 MVP

## 设计摘要

采用纯 Markdown + Git 的单仓库模式，核心规范和知识全部放在仓库中，工具差异放入 `adapters/`。`AGENTS.md` 只作为入口索引，详细规范分层存放在 `context/`、`requirements/`、`skills/`、`agents/`、`commands/`。

## 上下文依据

| 依据 | 来源 |
| --- | --- |
| 轻入口、按需读取 | `AGENTS.md` |
| 团队和项目公共记忆 | `context/INDEX.md` |
| 需求生命周期载体 | `requirements/README.md` |

## 方案

### 数据流

```text
用户需求 -> AGENTS.md -> context 索引 -> requirements 产物 -> 门禁检查 -> notes.md -> context 沉淀
```

### 模块改动

| 模块 | 改动 | 风险 |
| --- | --- | --- |
| 根目录 | 增加人类入口和 AI 入口 | 入口文档膨胀 |
| context | 增加团队、项目、经验三层知识结构 | 知识过时 |
| requirements | 增加需求生命周期模板和示例需求 | 流程过重 |
| skills/agents/commands | 增加轻量能力定义 | 过早抽象 |
| adapters/eval | 增加工具适配和人工回归用例 | 后续维护不足 |

### 接口和契约

本 MVP 的接口是文件契约，不定义运行时 API。核心契约包括：

- `AGENTS.md` 是 AI 入口。
- `context/*/INDEX.md` 是上下文路由入口。
- 每个需求目录必须包含 7 个生命周期文件。
- 工具适配不得改变核心仓库规范。

## 兼容性

- 任何能读取仓库文本文件的 AI 编程工具均可使用。
- 工具专有能力只能写在 `adapters/`，不能污染核心规范。

## 风险和回滚

| 风险 | 影响 | 应对 | 回滚 |
| --- | --- | --- | --- |
| 规范过多 | 新人难以上手 | README 保留最小路径 | 删除低频规则 |
| 知识过时 | Agent 输出错误 | PR review 和月度瘦身 | 标记废弃或删除 |
| 过早工具化 | 维护成本上升 | MVP 限制 4 个 Skill | 合并或删除低频 Skill |

## 安全和隐私

- 是否涉及认证/授权：否。
- 是否涉及用户输入或文件上传：否。
- 是否涉及敏感数据：否。
- 是否需要安全审查：需要轻量审查，确认模板不包含密钥、本地私有路径或敏感资料。
