# Tooling Governance

本规范用于决定什么时候新增或修改 Command、Skill、Agent、Rule、Hook、Prompt 和 MCP 配置，避免模板在真实使用后膨胀失控。

## 分层职责

| 层级 | 作用 | 自主权 |
| --- | --- | --- |
| 设计原则 | 为什么这样做，稳定方向 | AI 只能建议，人类确认 |
| 能力预期 | 系统应该支持什么 | AI 可起草，人类确认 |
| 场景规范 | Agent 必须遵守什么 | AI 可修改，建议 review |
| 工具产物 | Command、Skill、Agent、Hook 等实现 | AI 可修改，必须验证 |

## 是否应该工具化

满足以下任意两项，才考虑新增 Skill、Agent、Command 或 Hook：

- 高频重复。
- 强制约束，不能靠人记。
- AI 经常遗漏或误用。
- 需要跨会话复用。
- 需要稳定输出格式。
- 可通过清单或脚本验证。

不应工具化：

- 一次性任务。
- 高度依赖人类业务判断的决策。
- 简单自然语言即可完成的低成本操作。
- 没有复用频率的数据或规则。

## 形态选择

| 形态 | 适用场景 | 约束 |
| --- | --- | --- |
| Command | 高频固定入口 | 只做预检和委托，避免复制 Skill 逻辑 |
| Skill | 需要领域知识、多步工作流或质量基准 | `SKILL.md` 保持轻量，详细资源一跳可达 |
| Agent | 大量输出、独立探索、隔离上下文 | 只返回摘要，不嵌套调用其他 Agent |
| Rule | 跨任务稳定约束 | 放入 `rules/`，按通用/语言分层 |
| Hook | 必须机械化拦截或提醒的约束 | 支持禁用、分级、跨平台说明 |
| MCP | 外部系统连接能力 | 不包含密钥，只保留配置模板和风险边界 |

## 变更检查清单

- [ ] 是否更新了相关 `INDEX.md` 或 README。
- [ ] 是否说明了触发场景和不适用场景。
- [ ] 是否避免复制已有规则。
- [ ] 是否增加或更新了 eval 用例。
- [ ] 是否记录了设计原因。
- [ ] 是否评估了上下文成本。

## 定期 stocktake

每月执行一次工具盘点：

- 触发频率。
- 成功率。
- 拦截到的问题数。
- 误报或噪音。
- 是否有更简单的替代方式。
